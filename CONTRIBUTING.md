## 你不需要会写代码

这个项目最核心的参与方式，是**提需求**。

你生活中有没有遇到过这种时刻："要是有个工具能帮我做这件事就好了"？把这个想法说出来，就是最大的贡献。好的产品从来不是程序员想出来的，是用户想出来的。

### 怎么提需求？

**[👉 点这里，填个表就行](https://github.com/openvibelab/openvibelab/issues/new?template=project-request.yml)**

说清楚三件事：

1. **你想要什么** — 一句话描述
2. **为什么想要** — 解决你的什么问题
3. **最好长什么样** — 有参考就贴，没有也完全没关系

提交之后：

1. 我会回复确认收到
2. 评估可行性，标记状态
3. 社区投票，票多的优先
4. 排期开发

### 怎么让你的需求更快被做？

说白了就是三件事：**描述清楚**（让我秒懂你要什么）、**拉人投票**（让更多人给你的 Issue 点 👍）、**积极互动**（我问你问题时及时回复）。

## 你会写代码？那更好了

**每个项目都欢迎 PR。**

修一个 typo 是贡献，加一个功能是贡献，重构一段烂代码也是贡献。不用客气，直接来。

### 怎么提 PR

1. Fork 对应的项目仓库
2. 创建分支：`git checkout -b feature/你的功能`
3. 写代码，提交：`git commit -m "feat: 添加了xxx"`
4. 推送：`git push origin feature/你的功能`
5. 创建 Pull Request，写清楚改了什么

### 什么样的 PR 容易被合并？

- **解决真实问题** — 不是为了改而改
- **改动聚焦** — 一个 PR 做一件事
- **代码清晰** — 别人能看懂
- **有说明** — PR 描述写清楚 why，不只是 what

### Commit 规范

```
feat:     新功能
fix:      修 bug
docs:     文档
style:    格式调整
refactor: 重构
test:     测试
chore:    工具/构建
```

## 🚀 好项目会被做大

100 天挑战里的项目，大部分是 1-3 天快速搞出来的 MVP。这是故意的 — 先验证想法，再决定投入。

但如果一个项目：

- 用的人多，反馈好
- 确实解决了一个真问题
- 有社区成员愿意一起维护

**我们会把它升级为正式项目。** 不再是 demo，而是一个认真打磨的产品。功能完善、长期维护、上线小程序和 App。

参与过贡献的人，自然是核心团队。**你的 PR 不会白提。**

## 反馈

用了某个项目之后，去对应的仓库提 Issue：

- 🐛 **发现 Bug** — 告诉我哪里坏了
- 💡 **功能建议** — 觉得还能加什么
- 📝 **使用反馈** — 好用就夸，不好用就骂（友善地）

## 社区公约

简单四条：

- 尊重每一个人
- 友善地表达不同意见
- 不攻击、不歧视
- 帮助新人
